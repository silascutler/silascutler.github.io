Love And Hate With JARM

Before everyone in the US went on holiday for Thanksgiving, I spent some time experimenting with a new tool from Salesforce - JARM. As described in their blog post: “JARM is an active Transport Layer Security (TLS) server fingerprinting tool“. Despite my initial excitement, it is my current opinion that JARM fingerprints alone are rarely (not always) unique enough to be a reliable method for clustering. When using JARM hashes, it’s critical to remember that you’re fingerprinting the TLS settings only, not the application or the server.

The following section will outline the testing I have done.

Experiments

While Shodan would be the logical place to start (because of their early implemented JARM scanning/searching), I manage to burn through my API limits by 9AM. Instead, I opted to just scan the entire Internet so I could explore the raw data without API limits to constraint my curiosity.

In the Salesforce blog for JARM, they note it can be used to:

  • Quickly verify that all servers in a group have the same TLS configuration.
  • Group disparate servers on the internet by configuration, identifying that a server may belong to Google vs. Salesforce vs. Apple, for example.
  • Identify default applications or infrastructure.
  • Identify malware command and control infrastructure and other malicious servers on the Internet.

In lieu of proper Scientific Methodology, in which I would create a series of hypotheses and test the validity of each, I opted to break these use cases into how well does JARM do at Clustering and Hunting.

Final Thoughts
As a stand-alone method for identifying common infrastructure, JARM hashes are an unreliable method.

When using JARM, it is critical to remember, you are not fingerprinting the TLS settings - not the end application. If you are hoping to use JARM to identify malware control servers that have web-based panels, JARM will let you down, except in some rare cases.
